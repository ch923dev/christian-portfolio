Overview

The problem worth solving, and who it was for.

The problem

Support teams drown in untriaged tickets. Every conversation lands in one queue, and a human has to read, categorize, and route it before the real work even starts. At volume, that triage tax delays responses and burns out the team.

The Triage App reads each incoming Intercom ticket, understands it against the product’s own knowledge base, and tags + routes it automatically, so humans start at the answer, not the sorting.

“Don’t make a person do what a grounded model can do in a second — let them spend that second being helpful.”

Approach

Key product and engineering decisions.

Grounded, not guessing

Generic classification hallucinates categories that don’t match the product. I built the agent on retrieval-augmented generation: the knowledge base is embedded into a vector store, and every ticket is classified against the company’s real documentation, not the model’s assumptions.

Confidence-aware routing

The agent returns a category and a confidence score. High-confidence tickets route automatically; ambiguous ones are flagged for a human with the model’s reasoning attached, so trust builds instead of breaking.

  • RAG pipeline grounding every decision in the product KB
  • Auto-tagging with category + confidence on each ticket
  • Instant routing to the correct team via the Intercom API
  • Human-in-the-loop fallback for low-confidence cases
How it works

The shape of the system, at a glance.

The agent is a grounded classifier. Rather than guessing categories, it reasons about each ticket against the product’s own knowledge base and returns a category with a confidence score: high-confidence tickets route themselves, ambiguous ones go to a human with the reasoning attached.

The design choice that matters: the knowledge base is the source of truth. Improving triage means improving the docs: no retraining, no brittle rule trees to maintain as the product changes.
